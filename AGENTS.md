# Agent guide

Context for AI coding assistants (and humans new to the repo) working on **Zekr**, a test framework for ReScript.

## What Zekr is

Zekr is a small, zero-runtime-dependency test framework for ReScript. It provides:

- Test and suite factories with skip/only modes and lifecycle hooks
- A rich assertion library
- A CLI runner with filtering, watch mode, and timeout support
- Snapshot testing
- A jsdom-backed DOM testing module inspired by Testing Library

The framework is distributed on npm as `zekr` and is consumed by other ReScript projects as a dev dependency.

## Repository layout

```
src/              Library source (ReScript)
tests/            Tests for the library itself (dev-only sources)
docs-website/     Vite + ReScript documentation site, published to GitHub Pages
scripts/          Build helpers (sourcemap generator for coverage)
README.md         User-facing documentation
CHANGELOG.md      Release notes (generated by semantic-release)
rescript.json     ReScript build config (namespace: true, in-source ESM)
package.json      npm package metadata; "main" points at src/Zekr.js
```

### `src/` modules

Every `.res` file in `src/` becomes a public submodule under the `Zekr`
namespace. The library is split along functional lines:

| File              | Public name         | Responsibility                                      |
| ----------------- | ------------------- | --------------------------------------------------- |
| `Types.res`       | `Zekr.Types`        | Shared types (`testResult`, `testCase`, `testSuite`, hooks) |
| `Test.res`        | `Zekr.Test`         | Test case factories                                 |
| `Suite.res`       | `Zekr.Suite`        | Suite factories with hooks                          |
| `Assert.res`      | `Zekr.Assert`       | Assertions returning `testResult`                   |
| `Runner.res`      | `Zekr.Runner`       | Run suites, filtering, watch mode, timeouts         |
| `Snapshot.res`    | `Zekr.Snapshot`     | File-backed snapshot testing                        |
| `Colors.res`      | `Zekr.Colors`       | Terminal color helpers                              |
| `DomTesting.res`  | `Zekr.DomTesting`   | `render`/`cleanup` + `.Query`/`.Event`/`.Assert`    |
| `DomBindings.res` | `Zekr.DomBindings`  | Low-level jsdom bindings                            |
| `DomQuery.res`    | `Zekr.DomQuery`     | `getBy*`/`queryBy*`/`getAllBy*` query helpers       |
| `DomEvent.res`    | `Zekr.DomEvent`     | User-event simulation                               |
| `DomAssert.res`   | `Zekr.DomAssert`    | DOM assertions (`toBeVisible`, `toHaveValue`, …)    |

`DomTesting` is the entry point for DOM tests; it re-exports the three
companion modules as `DomTesting.Query`, `DomTesting.Event`, `DomTesting.Assert`
so users only need one module in scope.

## Using Zekr in a ReScript project

Add to consumer's `rescript.json`:

```json
{ "dev-dependencies": ["zekr"] }
```

### Writing a test

```rescript
open Zekr

let arithmeticSuite = Suite.make("arithmetic", [
  Test.make("adds", () => Assert.equal(1 + 1, 2)),
  Test.make("is positive", () => Assert.isTrue(1 + 1 > 0)),
])
```

Test files define suites with `Suite.make` / `Suite.async` and do **not** call
`Runner.runSuites` or `Runner.runAsyncSuites`. The zekr CLI discovers every
`*.test.res` file and runs its registered suites automatically via
`bin/zekr-run.mjs`.

`Runner.runSuites` / `Runner.runAsyncSuites` remain available for
manual/standalone execution (e.g. running a compiled test file directly with
`node`), but are not needed when using the CLI.

### Test factories (`Zekr.Test`)

```rescript
Test.make("name", () => Pass)                  // normal
Test.skip("wip", () => Pass)                   // skipped at runtime
Test.only("focus", () => Pass)                 // if any `only` test exists, only those run
Test.async("fetches", async () => Pass, ~timeout=5000)
Test.asyncSkip("not yet", async () => Pass)
Test.asyncOnly("just this", async () => Pass)
```

A test function returns a `testResult` (`Pass` or `Fail(string)`). Assertions
return `testResult`, so the simplest test just returns an assertion directly.

### Suites and hooks (`Zekr.Suite`)

```rescript
Suite.make(
  "db", 
  [test1, test2],
  ~beforeAll=() => initDb(),
  ~afterAll=() => closeDb(),
  ~beforeEach=() => clearTables(),
  ~afterEach=() => resetState(),
)

Suite.async(
  "api", 
  [asyncTest1, asyncTest2],
  ~beforeAll=async () => await connect(),
  ~afterEach=async () => await resetMocks(),
)
```

All hooks are optional. Async hooks are available only on async suites.

### Assertions (`Zekr.Assert`)

Every assertion returns `testResult` and accepts an optional `~message`:

```rescript
Assert.equal(actual, expected)
Assert.notEqual(actual, expected)
Assert.isTrue(cond)
Assert.isFalse(cond)

Assert.greaterThan(a, b)
Assert.lessThan(a, b)
Assert.greaterThanOrEqual(a, b)
Assert.lessThanOrEqual(a, b)

Assert.contains(haystack, needle)           // string
Assert.arrayContains(arr, item)
Assert.matches(str, regex)                  // RegExp.t

Assert.some(optValue)
Assert.none(optValue)
Assert.ok(resultValue)
Assert.error(resultValue)
Assert.throws(() => risky())

Assert.combineResults([r1, r2, r3])         // all-or-first-fail
Assert.combineAsyncResults([p1, p2])
```

Reserved-word aliases: `isTrue`, `isFalse`, `matches` — because `true`,
`false`, and `match` are keywords in ReScript.

### Running suites (`Zekr.Runner`)

```rescript
Runner.runSuite(oneSuite)                    // sync, single suite
Runner.runSuites([s1, s2])                   // sync, many suites

await Runner.runAsyncSuite(asyncSuite)
await Runner.runAsyncSuites([a1, a2])

await Runner.runWithTimeout(() => slow(), Some(5000))
```

Under the CLI, test files do not call `Runner.runSuites` or
`Runner.runAsyncSuites` directly — the harness (`bin/zekr-run.mjs`) imports
each compiled test file and calls `Runner.run()` to execute its registered
suites. `npm test` builds, then runs the `zekr` CLI, which discovers every
`*.test.res` file and spawns each compiled sibling in its own Node process via
the harness.

`Runner.runSuites` / `Runner.runAsyncSuites` remain available for
standalone execution: `node tests/MyTests.js` still works if the file ends
with a runner call.

#### Filtering

Runtime filtering via env vars:

```bash
ZEKR_FILTER="user" node tests/MyTests.js
ZEKR_SKIP="slow"  node tests/MyTests.js
ZEKR_FILTER="api" ZEKR_SKIP="integration" node tests/MyTests.js
```

Or programmatically:

```rescript
Runner.setFilterPattern(Some("user"))
Runner.setSkipPattern(Some("slow"))
```

Matching is case-insensitive and applies to both suite and test names.

#### Watch mode

```rescript
Runner.watchMode(
  ~testCommand="node tests/MyTests.js",
  ~watchPaths=["src", "tests"],
  ~buildCommand="npx rescript",
)
```

### Snapshots (`Zekr.Snapshot`)

```rescript
Snapshot.setDir("tests/__snapshots__")        // default: "__snapshots__"

Test.make("user shape", () => {
  Snapshot.matches(userValue, ~name="user-data")
})

Snapshot.update(newValue, ~name="user-data")  // programmatic update
```

On first run, a snapshot file is written. Subsequent runs compare against it.

### DOM testing (`Zekr.DomTesting`)

`DomTesting` boots a jsdom document and exposes Testing-Library-style helpers.

```rescript
open Zekr

Test.make("login form", () => {
  let {container} = DomTesting.render(`
    <form>
      <label for="email">Email</label>
      <input id="email" type="email" />
      <button type="submit">Sign In</button>
    </form>
  `)

  let input = container->DomTesting.Query.getByLabelText("Email")
  DomTesting.Event.typeText(input, "a@b.com")

  let result = Assert.combineResults([
    DomTesting.Assert.toHaveValue(input, "a@b.com"),
    DomTesting.Assert.toBeEnabled(
      container->DomTesting.Query.getByRole("button"),
    ),
  ])

  DomTesting.cleanup()
  result
})
```

Queries come in three variants per strategy:

- `getBy*` — returns the element; throws if not found or ambiguous
- `queryBy*` — returns `option<Dom.element>`
- `getAllBy*` — returns `array<Dom.element>`; throws if none

Available strategies: `Text`, `Role`, `TestId`, `Placeholder`, `LabelText`,
`AltText`, `Title`, `DisplayValue`.

Events (`DomTesting.Event.*`): `click`, `dblClick`, `hover`, `unhover`,
`typeText`, `clear`, `check`, `uncheck`, `selectOptions`, `focus`, `blur`,
`fire` (low-level dispatch).

Assertions (`DomTesting.Assert.*`): `toBeInTheDocument`, `toHaveTextContent`,
`toHaveAttribute`, `toHaveClass`, `toBeVisible`, `toBeDisabled`, `toHaveValue`,
`toBeChecked`, `toContainElement`, `toContainHTML`, `toBeEmptyDOMElement`,
`toHaveStyle`, `toHaveFocus`, and their negated `toNot*` counterparts.

Always call `DomTesting.cleanup()` at the end of a DOM test (or from an
`~afterEach` hook) to remove rendered containers between runs.

## Module namespacing

`rescript.json` sets `"namespace": true`. ReScript derives the namespace from
the package name (`zekr` → `Zekr`) and wraps every top-level `.res` file in
`src/` as a submodule of a synthetic `Zekr` module. Consumers see a single
`Zekr` module whose members are the `.res` files.

Implications:

- Do **not** create a `Zekr.res` facade — it collides with the auto-generated
  namespace module.
- Adding a new `.res` file to `src/` automatically makes it `Zekr.<NewName>`.
- Inside the package, modules are referenced by their bare name
  (`Test.make`, `Runner.runSuite`, etc.). The `Zekr.` prefix only appears in
  external consumers.

### Why `DomTesting` and not `Dom`?

`DomAssert`, `DomBindings`, `DomQuery`, and `DomEvent` all reference
`Dom.element` and `Dom.event` from the ReScript stdlib. A local `Dom.res`
would shadow stdlib `Dom` throughout the package and break those references.
`DomTesting.res` sidesteps the collision.

## CLI (`zekr` binary)

- `src/Cli.res` (`Zekr.Cli`) holds all discovery and execution logic. Pure
  helpers (`parseArgs`, `resolveConfig`, `parseFileConfig`, `candidateSiblings`)
  are kept separate from IO helpers (`readFileConfig`, `runFiles`, `main`) so
  the pure layer remains unit-testable.
- `bin/zekr.mjs` is a shebang shim (`#!/usr/bin/env node`) that imports the
  compiled `src/Cli.js` and delegates to `main`.
- `bin/zekr-run.mjs` is the per-file harness: it imports one compiled test
  file (registering its suites) and calls `Runner.run()`. It resolves the
  Runner module through `bin/resolve-runner.mjs`, which picks the
  `Runner<suffix>` matching the test file's compiled suffix (`.res.mjs`,
  `.mjs`, `.bs.js`, `.js`, …). This is essential: ReScript compiles the whole
  build graph — zekr's sources included — with the consumer's `suffix`, so the
  harness must load the Runner sharing the same `Registry` instance the suites
  registered into. Loading a mismatched Runner reads an empty registry and
  reports zero tests (a false green). `resolveRunnerUrl` is pure and
  unit-tested (`tests/Harness.test.res`); it falls back to `Runner.js` when no
  suffix-matched Runner sits next to the harness.
- **`Zekr.Cli` is deliberately NOT re-exported from `src/Zekr.js`.** It is the
  CLI entry point, not a library API. This is an intentional, documented
  exception to the barrel rule below — do not add it to the barrel.
- Default suffix is `.test.res`; precedence is flag > `zekr.json` > default.
  Each matched file is spawned in its own Node process. The process exits
  non-zero on any test failure or if zero files match. `node_modules`, `lib`,
  `.git`, and dot-directories are skipped.
- `--watch`/`-w` runs an initial full pass, then watches the scan directory
  and re-runs only the tests impacted by each change. Impact analysis lives in
  pure helpers (`impactedTestFiles`, `moduleNameOfPath`, `toSourcePath`,
  `referencesModule`) so it stays unit-testable: a changed test file impacts
  only itself; a changed source module impacts every test that references it
  (`Module.` usage or `open Module`). The watcher reacts to compiled output
  (not raw `.res` saves) and debounces bursts, so run `rescript -w` alongside.

## The `src/Zekr.js` barrel

`namespace: true` does not emit a runtime JS file for the namespace module,
but `package.json` has `"main": "./src/Zekr.js"`. A small hand-written barrel
re-exports every submodule so plain JS/TS consumers can write:

```js
import { Test, Suite, Assert, Runner, DomTesting } from "zekr"
```

The barrel is checked into git via a `!src/Zekr.js` exception in
`.gitignore`. **If you add or remove a top-level module in `src/`, update the
barrel as well.** ReScript consumers do not need the barrel; they reach the
submodules through the synthesized `Zekr` namespace at compile time.

## Development workflow

```bash
npm install
npx rescript build        # one-shot build
npx rescript -w           # watch build
npm test                  # build, then run the zekr CLI over *.test.res
npm run coverage          # c8 coverage report with .res sourcemaps
```

Tests in `tests/` are dev-only sources declared under `"type": "dev"` in
`rescript.json`. They use the library API the same way external consumers do
(minus the `Zekr.` prefix, since they live in the same package). Each suite
file is named `*.test.res` so the `zekr` CLI discovers it; avoid naming one
`Zekr.test.res`, since that base name collides with the `Zekr` namespace.

### Docs website

```bash
cd docs-website
npm install
npx rescript build
npm run dev               # Vite dev server
npm run build             # static build for GitHub Pages
```

Pages live in `docs-website/src/pages/Pages__*.res`. Each API page typically
mirrors one library module. When you add or rename a public API, update both
the README and the corresponding page.

## Conventions for changes

- **Keep the public module surface stable.** Renaming or removing a
  top-level `.res` file in `src/` is a breaking change; bump the major.
- **Keep function names free of module-name noise.** e.g. `Assert.equal`,
  not `Assert.assertEqual`; `Snapshot.matches`, not
  `Snapshot.assertMatchesSnapshot`.
- **Assertions return `testResult`**, never throw. If a helper needs to
  signal a failure, return `Fail(msg)` with a human-readable message that
  shows the actual and expected values.
- **DOM tests must call `DomTesting.cleanup()`** — enforce it locally in a
  test or globally via an `~afterEach` hook.
- **Update the JS barrel** (`src/Zekr.js`) whenever you add or remove a
  top-level module.
- **Update docs alongside code.** README, AGENTS.md (this file), and the
  matching `docs-website/src/pages/Pages__Api*.res` page should stay in sync
  with the API.
- **Run `npm test` before committing.** All library tests must pass; the
  framework dog-foods itself.
- **Commits follow Conventional Commits.** `semantic-release` drives the
  version bump and changelog from commit messages. Use `feat:`/`fix:` for
  user-visible changes and add `!` (or a `BREAKING CHANGE:` footer) for
  breaking ones.
